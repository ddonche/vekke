Vekke Game Notation

By Intl. Vekke Council ~ 8 min read

Vekke Game Notation (VGN) — v1

This explains the Vekke Game Notation system.

Core Principle

VGN is a pure state-transition ledger.

Each non-META line records only factual state changes:

  • time (t)
  • player (p, if applicable)
  • an object (token or route)
  • a transfer from one location to another

Game semantics are never encoded as events.

All named game concepts (invade, siege, evasion, mulligan, draft, early swap, etc.) appear only in NOTE lines.

This guarantees:

  • full replayability
  • rule-agnostic logs
  • long-term stability
  • zero semantic drift

Line Format

Each line is a single state change or annotation:

t=<ms>|key=value|key=value|...

Rules

  • t = milliseconds since game start
  • Fields are pipe-delimited (|)
  • Field order is not significant unless stated
  • No whitespace unless inside quoted values
  • to= is always used for transfers
  • Coordinates (A1–F6) imply BOARD

Player Field (p)

  • For any line that mutates a player-owned object (tokens, hands), p is required.
  • For global/system operations that are not attributable to a player (ex: queue refill from the deck), p is omitted.

Optional Readability Fields

These fields do not change state and may be included on any non-META line:

  • r=<n> — the current round number (ties events to the most recent ROUND|n=<n> marker)
  • dt=<ms> — elapsed milliseconds since the previous non-META line (redundant with t, provided for human scanning)

Canonical Locations

BOARD (implicit via coordinates)
RESERVE
CAPTIVE
VOID
HAND
QUEUE
DECK

Player-Scoped vs Global Locations

  • Player-scoped locations exist per player and are always interpreted in the context of p=:

    • RESERVE
    • CAPTIVE
    • HAND

  • Global locations are shared and have no player ownership:

    • BOARD (implicit)
    • QUEUE
    • DECK
    • VOID

Rule: When a player-scoped location is used, p must be present, and the location refers to that player’s pool.

Example:

t=...|p=B|from=E5|to=CAPTIVE

This always means: the token is moved into Blue’s CAPTIVE pool (the capturer).

Notes:

  • BOARD is never written explicitly
  • Any grid coordinate implies BOARD
  • All non-board locations are written verbatim

Tokens (Critical)

TOKENS HAVE NO IDS.

Tokens are indistinguishable and tracked only by:

  • owner (p=W or p=B)
  • location (coordinate or named location)
  • counts (when not on board)

Therefore:

  • Board interactions reference coordinates
  • Off-board interactions reference counts

Routes

Routes are immutable values written as:

dir/dist
  • dir ∈ 1–8 (compass order, clockwise from North)
  • dist ∈ 1–4
  • Orthogonal directions may use distance 4

Routes move between:

  • HAND
  • QUEUE
  • DECK

Routes are always transferred explicitly between locations.

Required Header (META)

These lines must appear before gameplay:

META|GAME|id=<uuid>|ruleset=<name>|version=1
META|PLAYERS|W=<player_id>|B=<player_id>
META|TOKENS|W=18|B=18
META|TC|mode=<standard|rapid|blitz>|base=<ms>|inc=<ms>

Round Marker

Rounds are explicitly marked:

t=...|ROUND|n=1

This line has no state effect and exists for:

  • turn structure
  • clocks
  • analysis

State Transfers

Board Placement

Opening placement uses the same explicit transfer format as reinforcement:

t=...|p=W|from=RESERVE|to=A3

Notes:

  • Opening placement is no longer implicit
  • This keeps all placements uniform and removes special cases

Board Movement

t=...|p=W|from=A3|route=6/1|to=A2

Notes:

  • Path is never logged
  • There is no MOVE event

Capture (Board Removal)

t=...|p=B|from=E5|to=CAPTIVE

Meaning:

  • Token at E5 is removed from board and moved to CAPTIVE

Game meaning (optional annotation):

t=...|NOTE|text="Siege at E5"

Reinforcement

t=...|p=W|from=RESERVE|to=C3

Yield (Generic)

All costs are expressed as explicit transfers to VOID, and the source location is always written:

t=...|p=W|from=RESERVE|to=VOID|yield=4

Rules:

  • yield is used only for transfers whose destination is VOID.
  • The yield source is never implied; it is always expressed via from=<LOCATION>.

Canonical yield meanings (early swap vs reinforce vs evasion, etc.) are determined by the ruleset and may be annotated with NOTE, but the VGN line remains a plain transfer.

Route Transfers

Queue to Hand

t=...|p=W|route=3/2|from=QUEUE|to=HAND

Hand to Deck

t=...|p=W|route=1/4|from=HAND|to=DECK

Deck to Queue (Refill)

Queue refill is a global/system transfer (no player attribution):

t=...|route=5/1|from=DECK|to=QUEUE

Early Reinforcement (State Only)

Early reinforcement is expressed only as state transitions:

t=...|p=W|from=RESERVE|to=C3
t=...|p=W|from=RESERVE|to=VOID|yield=4

Optional annotation:

t=...|NOTE|text="Early reinforcement"

Early Route Swap (State Only)

Early route swap is expressed only as state transitions:

t=...|p=B|route=7/2|from=QUEUE|to=HAND
t=...|p=B|route=1/4|from=HAND|to=DECK
t=...|p=B|from=CAPTIVE|to=VOID|yield=4

Optional annotation:

t=...|NOTE|text="Early route swap"

Mulligan (State Only)

t=...|p=W|from=D4|to=RESERVE
t=...|p=W|route=1/2|from=HAND|to=DECK
t=...|p=W|route=6/2|from=DECK|to=HAND

Optional annotation:

t=...|NOTE|text="Mulligan"

Draft (Void Recovery)

Draft is not a yield (it is recovery). Use count for non-VOID multi-token transfers:

t=...|p=W|from=VOID|to=RESERVE|count=2

Optional annotation:

t=...|NOTE|text="Draft"

Evasion (State Only)

t=...|p=B|from=RESERVE|to=VOID|yield=2
t=...|p=B|from=CAPTIVE|to=VOID|yield=2
t=...|p=B|from=D5|to=C6

Optional annotation:

t=...|NOTE|text="Evasion"

Notes

Notes are human-readable annotations ignored by engines:

t=...|NOTE|text="White invaded E5"

Rules:

  • Notes never change state
  • Text must be quoted
  • Any game terminology belongs here

Game End

Game end is recorded explicitly and symmetrically, with both winner and loser identified.

WIN / LOSS

t=...|WIN|type=RESIGN|winner=B|loser=W
t=...|WIN|type=ELIMINATION|winner=W|loser=B
t=...|LOSS|type=TIMEOUT|winner=B|loser=W

Rules:

  • WIN and LOSS are terminal state markers

  • type is required and must be one of:

    • RESIGN
    • ELIMINATION
    • TIMEOUT

  • Both winner and loser must be specified explicitly

  • These lines do not move tokens or routes

  • Optional NOTE lines may further annotate context

Design Guarantees

  • Fully replayable from state alone
  • Rule-agnostic and future-proof
  • Deterministic across engines
  • Human-readable without sacrificing rigor
  • No token IDs, no semantic coupling
